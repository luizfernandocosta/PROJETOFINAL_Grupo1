===== PAGE 1 =====
 Guia do Projeto Final – Fundamentos de Engenharia de Dados 
 
A cada grupo será atribuído um domínio de atuação: Saúde, E -commerce, Logística, 
Finanças, Educação, Agro, Mobilidade Urbana. O projeto deve aplicar o mesmo 
arcabouço técnico independentemente da origem dos dados. O storytelling e as 
perguntas de negócio serão definidos por cada grupo, respeitando os requisitos 
funcionais comuns descritos abaixo. 
A base de dados selecionada NÃO pode ser nenhuma utilizada previamente pelos 
membros do grupo nos laboratórios anteriores.  
 
Storytelling (Problema de Negócio): 
O grupo deve construir uma história, cenário hipotético. E deve definir algumas 
perguntas para uma área de negócio que precisa de um painel confiável e atualizado 
diariamente. Perguntas como: 
• Quais produtos geram mais receita e também mais devoluções? 
• Qual o tempo médio de entrega por região? 
• Existem pedidos com valores inconsistentes (frete maior que o total do 
pedido)? 
 
Desafio: 
O desafio do grupo é construir um pipeline ELT completo, avaliando a qualidade dos 
dados desde a ingestão (Raw), passando por transformações confiáveis (Silver/Gold), 
até a visualização. Tudo orquestrado e com rastreabilidade de linhagem. 
 
Fluxo: 
1. Airbyte → carrega os dados do dataset escolhido para o schema raw do 
PostgreSQL. (OPCIONAL) 
2. Script Python Customizado: Desenvolver um script Python para realizar a 
extração e carga (EL) dos dados brutos para o PostgreSQL. 
3. Great Expectations → valida os dados recém-chegados na raw (qualidade na 
entrada). 
4. dbt → transforma raw → silver (limpeza, tipagem) → gold (modelo estrela). 
5. Airflow → orquestra as etapas com tratamento de falhas. 
6. Ferramenta de visualização → conecta-se ao schema gold para dashboards. 
7. Docker → para reprodutibilidade.

        Extra 2 – Infraestrutura como Código com Terraform 
+ 10% na disciplina 
Os grupos que optarem por este extra deverão  provisionar todo o ambiente  (ou parte 
crítica dele) utilizando  Terraform, em vez de apenas  docker-compose. O objetivo é 
demonstrar capacidade de definir infraestrutura declarativa, mesmo que em ambiente 
local, containers Docker gerenciados pelo provedor docker do Terraform. 
 
Requisitos obrigatórios para o extra: 
• Arquivos .tf organizados (provedores, variáveis, recursos, outputs). 
• Uso de variáveis (ex.: postgres_password, airflow_user) 
com terraform.tfvars.example. 
• Outputs claros (URLs dos serviços, comandos para acessar logs). 
• O pipeline de dados (Airbyte → Great Expectations → dbt) não precisa ser 
disparado pelo Terraform – o Terraform cuida apenas da infraestrutura estática. 
A orquestração continua sendo do Airflow.
